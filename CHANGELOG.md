# Changelog

## 0.16.0 - 2026-07-10
- **@abfcode/spine 0.9 → 0.11.** Two upstream releases in one bump, zero adaptation needed (all 140 tests and the real-book corpus pass unchanged). 0.10 is spine's pre-1.0 hardening release: DEFLATE-bomb ceilings on zip inflation (a hostile EPUB degrades instead of exhausting memory), ruby annotations handled correctly for CJK books (furigana readings dropped from extracted text, so search sees 漢字 not 漢かん字じ), and the .txt chapter splitter's weak rules gained consistency bounds — detected-break precision went from 6.9% to ~52% against a ground-truth oracle. 0.11's data-model pass makes the writer's simple metadata fields authoritative and teaches the cover to own its path (both of which our 0.13-era export workarounds already anticipated — they keep working and are now belt-and-suspenders), and fixes title/series extraction for books that list subtitles or non-series collections first. `PARSER_VERSION` tracks spine, so every library re-parses on next open and picks all of this up retroactively.
- **Password reset, at last.** "Forgot password?" on the sign-in page emails a reset link (new `/forgot-password` and `/reset-password` pages, anti-enumeration confirmation copy, expired-link recovery). Mail goes out as `hello@librium.dev` through Resend via the official `@convex-dev/resend` Convex component (durable queue, exactly-once), wired into Better Auth's `sendResetPassword` — plus email verification plumbing behind `REQUIRE_EMAIL_VERIFICATION` for when public signups open. Everything is env-gated on `RESEND_API_KEY` (absent = skipped with a logged warning, auth flows unaffected), and the reset endpoint gets its own rate limit (5/min) so it can't flood a victim's inbox. Live-verified end to end: request → emailed token → new password works, old refused.
- **Account deletion is one command.** `npx convex run admin:deleteUserAccount '{"email":"...","confirm":"DELETE"}' --prod` fulfills the privacy-page promise as a single operator-run action: every app row across all seven tables, the R2 objects (derived keys, so mid-import books can't strand blobs), and the Better Auth records (sessions, credentials, verifications, user) — with accounts that signed up but never imported handled too. The Polar customer is deliberately left to Polar (merchant-of-record retention obligations). Cross-contamination pinned by tests (a second user's rows survive untouched) and smoke-tested live: sign up → import → delete → the same email can immediately register fresh.

## 0.15.0 - 2026-07-09
- **Supporter subscriptions + cloud-storage quota (Polar).** Librium can now pay for itself: the free plan gets a cloud-storage allowance (default 250 MB of stored EPUBs — local reading is never limited), and a yearly Supporter subscription raises it (default 10 GB). Payments run through Polar as **merchant of record** — Polar is the seller and handles checkout, receipts, global VAT/sales tax, and refunds; Librium stores only the subscription status (synced by webhook into the Convex Polar component). New "Account & storage" dialog (library ⋯ menu) with a usage bar and checkout/customer-portal links; the import page shows usage while enforcement is on. **Data-safety rules, in priority order:** quota gates *new uploads only* — sync, seeding, reading, export, metadata edits, and deletes are never quota-checked, and a lapsed subscription locks nothing, ever. **Enforcement is on verified bytes, not client claims:** a new `finalizeUpload` action HEADs R2 for the object's true size and attaches inside a serializable mutation (concurrent imports can't race past the limit together); rejected uploads are deleted from the bucket and the local copy is untouched; `fileSize` now always records what R2 actually stores. Covers live outside the quota behind a 10 MB sanity cap. Everything is env-gated (`QUOTA_ENFORCED`, `POLAR_*`) and defaults to off/free, so the code deploys before the Polar org exists. Billing lookups fail open to the free plan — a Polar outage can never block reads or grant paid limits. New `/terms` and `/privacy` pages: plain-language, with an explicit wind-down clause (3 months' read-only notice + refunds + export always works) and a complete data inventory.
- **Review hardening (billing):** the 8-angle pre-release review confirmed 18 findings, 15 fixed before anything ships. The load-bearing ones: a finalize retry on an already-attached book could hit the quota-rejection path and **delete the book's live cloud EPUB** (now short-circuits to success; epub upload URLs also can't be re-minted once attached, making the master immutable); the quota was bypassable by uploading and simply never finalizing (public `api.r2.generateUploadUrl` removed, `deleteBook` deletes derived keys unconditionally, and a **daily orphan sweep** reclaims unclaimed `books/*` objects — accounting now converges to bucket reality); a Polar product created before the webhook existed made `getCurrentSubscription` throw, so a **paying supporter would have silently resolved to the free plan** (`billing:syncProducts` + loud fail-open logging); an oversized replacement cover destroyed the existing cover before validation (client refuses before the destructive fixed-key PUT); a quota-rejected import stranded a phantom metadata-only book on every device (now cleaned up); and a cover failure after the EPUB attached marked the import "failed", inviting duplicate re-imports (now a warning on the done item). Deferred knowingly: per-check usage summation (a denormalized counter risks drift in exactly the numbers that must stay honest), shared dialog-dismiss hook.
- **Adversarial + property tests for the quota plane** (`quota.convex.test.ts`, in CI): a fast-check property drives arbitrary interleavings of registers/attaches/deletes against the real Convex functions asserting the core invariant — verified attached bytes never exceed the limit, rejections change nothing, deletes free space — plus targeted adversarial cases: a client lying about file size is caught at finalize, non-finite/negative sizes rejected, re-finalize doesn't double-count, cross-user finalize refused, the legacy attach path refuses while enforcement is on (no unverified bypass), oversized covers rejected with the dangling key cleared, quota strictly per-user, and over-limit users keep full read/delete access. Proven falsifiable (sabotaged double-count and weakened limit both caught).

## 0.14.0 - 2026-07-09
- **Import .txt and .md webnovel rips.** Drop a raw text rip on the import page and it becomes a first-class book: spine 0.9.0's text ingestion (`@abfcode/spine/text`) converts it to a real EPUB — chapter detection via an ordered heuristic ruleset (Chapter N, roman numerals, CJK `第N章`, markdown headings, ALL-CAPS runs; biased to under-split), Gutenberg banner stripping (guarded), title from an authored markdown H1 or the filename — and then it rides the *entire* existing pipeline unchanged: R2 gets a proper EPUB, so seeding, sync, metadata editing, and export-with-write-back all work identically. Conversion runs in a module worker with the same in-page fallback shape as parsing. Verified live: the raw Project Gutenberg Alice `.txt` imports as a 29-chapter book. Bumping spine 0.8→0.9 also re-parses the library on next open (`PARSER_VERSION` tracks spine), though parser behavior is unchanged in 0.9.
- **Hardening pass.** A cleanup + testing + review sweep with no user-facing change. **Property-based sync tests** (fast-check over the real merge functions via convex-test, wired into CI) now guard the two documented bug classes across *arbitrary* operation orderings — the multi-device reality: status LWW always resolves to the highest-editedAt write regardless of apply order, and bookmark tombstones never resurrect (a key ever deleted stays tombstoned, same-key creates never duplicate). Both proven falsifiable. **Full-repo security review** came back clean: every mutation/query/action enforces ownership (read-URL signers, R2 key-prefix binding, collection/bookmark id-ops, the delete path), the metadata actions keep their SSRF guards (https-only, private-host block, per-hop redirect revalidation, size caps), admin/seed functions are env-gated off in production, no `dangerouslySetInnerHTML` anywhere, secrets are env-only, and the extension holds minimal permissions; `pnpm audit` clean. **Refactor:** the 1,969-line `ReaderExperience` shed its parsed-block renderer (`ReaderBlocks`) and link-resolution helpers (`readerLinks`) — behavior-identical, down to 1,616 lines. Plus the intermittent `library.spec` CI flake fixed at its root (an async-merge poll raced the ~1s default), `pwa.ts` split into its two concerns, shared test fixtures, and the codebase's last stale TODO retired.
- **Immersive reading on mobile.** The top bar no longer stays pinned — on phones it slides away as you read and returns when you scroll to the top or tap the page, so the text owns the screen. A tap anywhere (outside links/images/selection) toggles the chrome: the top bar plus a new **thumb-zone bottom bar** — ‹ Prev · chapter/progress · Next › where the center opens the chapter sheet. No tap-to-advance and no edge zones, so there are no accidental jumps; you scroll to read and tap to reveal navigation. The chapter sheet itself rises as a **bottom sheet** on phones (full-width, rounded top), auto-scrolled to your current chapter. Toggle decided on `click` not pointerup, so the tap's own synthetic click can't hit freshly-rendered chrome and undo itself.
- **Persistent-storage request moved to first download.** The browser's "keep this stored?" prompt (Safari eviction mitigation) now fires when a book's content first lands on the device — import or download-to-device — instead of at login, so the ask is self-explanatory. Denying it costs nothing functional: an evicted library just re-seeds from R2 on next open.
- **Review hardening (txt import):** the pre-release review confirmed and fixed 4 findings — `fileSize` now records the converted EPUB actually stored in R2 (not the source rip's size; `fileName` deliberately keeps the source name as provenance), spine's `stripBoilerplate` data-loss warnings surface to the console instead of firing into the void, the import page's empty-queue error stopped saying "EPUB file", and the accepted-extensions filter composes `isTextImport` instead of a third hand-rolled regex. Deferred knowingly: a shared worker-with-fallback helper (rule of three met, but unifying three proven worker paths is its own refactor) and the server-side title-fallback regex (unreachable — spine floors titles at "Untitled").
- **Book downloads got a real progress bar.** The reader's first-open seeding state ("Downloading book to this device…" as bare text) now streams the R2 download and shows an actual percentage — paragraph shimmer, a thin gold bar, "Downloading book — 42%", then "Preparing book…" while the parse runs. The heading shows the book's title instead of "Untitled chapter" while no section exists yet. Verified live under a throttled connection.

## 0.12.0 - 2026-07-07
- **@abfcode/spine 0.1.1 → 0.8.0.** Seven minor versions of the parser in one jump, adapted end to end: the reader's inline renderer moved to the 0.7.0 model (styling is combinable `emph`/`strong`/`code` flags, so `<strong><em>x</em></strong>` no longer loses a style; legacy-kind blocks still render until they re-parse), the ingest pipeline stopped re-resolving image srcs and link hrefs (spine ≥0.6 pre-resolves them to archive-relative paths — re-joining corrupted every image path, caught by a new real-EPUB corpus test), internal link resolution tries the pre-resolved href directly before falling back to relative resolution for old blocks, and search text extraction no longer injects a space around every styled run (`(sic)`, not `( sic )`). `PARSER_VERSION` now comes from spine's own `VERSION` export, so this bump — and every future one — automatically re-parses the whole library from raw EPUBs on next open, picking up 0.2–0.8's fidelity fixes (nested styling, malformed-XML recovery, entity handling, calibre series fallback) retroactively. Parsing is 5–7× faster upstream; the full 6MB/1,003-section memorize testbook parses in well under a second.
- **EPUB parsing moved off the main thread.** spine 0.4.0 dropped its DOMParser dependency, unblocking the roadmap's deferred Web Worker item: imports and device seeding/re-parse now run in a module worker (image/cover bytes transfer back zero-copy), so a 2,000-chapter webnovel import no longer freezes the UI. Environment failures degrade to a main-thread parse — the worker is an optimization, never a point of failure — and the degraded path logs a marker that the reading e2e asserts is *absent*, proving the worker really runs in a real browser (vitest's browser mode can't host Vite module workers, so unit tests cover the fallback and e2e covers the worker). The worker chunk rides the PWA precache, so offline seeding keeps working. Verifying this also caught a real bug: a parse failure inside the fallback path previously hung the import promise forever.
- **EPUB corpus tests** (`epubCorpus.test.ts`), two tiers: a synthetic image-bearing EPUB always runs (CI included) and pins the src-resolution invariant — every block-referenced image must resolve to extracted bytes (the regression that would have shipped without it); the real `testbooks/` novels additionally run on dev machines (structure pinned: 1,003 sections for memorize, 32 for worm4) but are copyrighted files kept out of the public repo, so those skip in CI.

## 0.11.0 - 2026-07-07
- **NovelUpdates clipper extension** (`extension/`): a one-click companion for the ~100x webnovel-linking flow — right-click → "Copy to Librium" (or the toolbar icon) on any NU series page copies the page HTML *and* the cover image as one JSON clipboard payload; pasting into the Edit-details dialog fills the source URL, opens the candidate diff, and stages the cover in one go. Getting the cover took three attempts, live-tested against real Cloudflare: extension background fetch is 403'd even with `credentials:"include"` (the clearance cookie is partition-bound to the tab), page-context fetch is CORS-sealed (CDN sends no `Access-Control-Allow-Origin`), so the shipped path **screenshots the rendered page and crops the cover** (`captureVisibleTab` + `OffscreenCanvas`) — no network request, nothing for Cloudflare or CORS to judge. That API demands the `activeTab` grant, which browsers hand out only for clicks on extension UI — hence context-menu/toolbar triggers and no injected button (a v0.1.x floating button could never capture; replaced by a status toast). Both doomed fetch paths were deleted after the live verdicts, so a successful run's console stays clean. Cross-browser MV3 (Chrome service worker + Firefox event page via dual background keys; `browser_specific_settings.gecko` set), host access scoped to novelupdates.com only, no server contact — clipboard out only. Versioned payload contract (`librium: 1`, diagnostic `coverError` on failure) parsed by `parseLibriumPayload` with per-field validation; unrecognized pastes fall through to normal text behavior. `parseNovelUpdatesHtml` is verified against a minimal, entirely synthetic selector-contract fixture.
- **Covers past Cloudflare:** NovelUpdates fronts its image CDN with the same Cloudflare wall as its pages (verified: `cdn.novelupdates.com` 403s every server-side fetch regardless of User-Agent), so the cover proxy could never fetch an NU candidate's cover and Save died with a bare 403 *after* the text fields had already committed. Three fixes: fetched-cover downloads now try the **user's browser first** (real browser traffic passes Cloudflare, and CORS-friendly hosts like Open Library / Google Books allow reading the bytes) with the server proxy as fallback; a cover failure is **non-fatal** — details stay saved and the dialog explains the way forward instead of a bare error; and you can now **paste an image (Ctrl+V)** anywhere in the Edit-details dialog to set the cover — right-click → Copy image on the NU page, paste, Save. No download-to-disk required.
- **Review hardening (clipper + cover pipeline):** an 8-angle review of the extension and dialog changes confirmed and fixed 6 findings. Highlights: applying an NU candidate's diff no longer clobbers a staged file cover (pasted/extension-captured) with the candidate's url cover — the cover row is checked by default and NU CDN urls are guaranteed to 403 at Save, so `applyFetched` now never downgrades file→url (falsifiable regression test included); the browser-first cover download enforces the same 4MB cap as the server proxy (an oversized direct hit previously uploaded uncapped); the extension's context menu survives Firefox restarts (removeAll+create on `onStartup`, not just `onInstalled`); a clipboard image item that yields no File no longer swallows a co-present payload paste; a truncated clipper payload is caught by prefix instead of dumping ~200KB of JSON into the focused field; and a payload whose cover fails to decode now says so instead of silently proceeding. Cleanups: the extension's two dead fetch paths (~120 lines, unreachable since every real NU cover lives on the blocked CDN) are deleted along with the now-unneeded `cdn.novelupdates.com` host permission, and the dialog's three copy-pasted pending-cover constructions collapsed into one `setCoverFromBlob`.
- **"Recent" sort no longer moves when you set a status.** Marking a book (or bulk-marking) bumps the userBooks sync clock — which the cross-device status merge legitimately needs — but that same clock also drove the default Recent shelf, so shelving an old book as Abandoned vaulted it to the top past what you were actually reading. Reading recency now lives in its own `lastActivityAt` field, written only by opening/reading a book (and at import), so status edits leave the shelf order alone. Never-opened books marked from the shelf sort last in Recent, as they should. New falsifiable e2e (`recentSort.spec.ts`) that fails against the old behavior — writing it also exposed that its first draft passed vacuously by racing the async status push, now synchronized.

## 0.10.0 - 2026-07-07
- **Edit details:** every book gets an editor (card menu → "Edit details…") — title, author, series/index, description, language, and a source-page URL; replace the cover from a local image. Saves only changed fields (empty = clear); server-authoritative and online-only, mirrored into the local shelf row so offline views agree. Replaced covers propagate to other devices via a `coverUpdatedAt` stamp (the R2 key never changes, so staleness is version-checked, on the server's clock).
- **Metadata fetch:** a "Fetch metadata" panel searches Open Library (no key) and Google Books (set `GOOGLE_BOOKS_API_KEY` on the Convex deployment to enable) — ISBN-first when the EPUB carried one, title/author otherwise. Candidates show covers and sources; picking one opens a per-field diff (current vs proposed, checkboxes) that merges into the form — **nothing touches the server until Save**. Fetched covers download through a server proxy (image hosts don't send CORS) with https-only/private-host/size guards, then ride the normal R2 + local-blob path.
- **NovelUpdates linking:** paste a series URL into the source-page field — "Open source page" everywhere it's shown, and "Fetch from linked page" pulls title/author/description/genres/cover from that exact page (no search matching). NU's Cloudflare blocks server fetches routinely, so the flow degrades to paste-the-page-HTML, parsed client-side into the same diff preview. The page-fetch action is allowlisted to novelupdates.com only.
- **Re-seed identity preservation:** re-downloading a book's content (device seeding, stale-parser refresh) no longer resurrects the EPUB's embedded metadata or cover over user edits — re-parse refreshes content, never identity. Covered by e2e (edit → remove download → re-download → edit intact).
- Reconcile now also mirrors drifted server metadata into local rows and invalidates stale local cover blobs; the R2 upload helper is shared (`uploadBookAsset`) between import and the editor.
- **Review hardening:** a 6-angle adversarial review (both 0.9.0 and 0.10.0 code) confirmed and fixed 18 findings across sync, security, and UI. Highlights: a collection membership un-checked during its create/add round-trip is no longer stranded alive on the server (dirty-flag stomp past a tombstone); status/collection merge writes now re-read the live row in a transaction so a concurrent edit can't be clobbered back to `dirty:0`; the metadata fetch actions re-validate **every redirect hop** (SSRF-via-redirect close), block IPv6 loopback/ULA/link-local/mapped hosts, and stream-abort oversized bodies instead of trusting `Content-Length`; `attachFiles` binds the R2 key to the book's own prefix (closes a cross-user object-read via a forged key); the Edit-details dialog diffs against its open-time snapshot (not the live doc) so a concurrent edit from another device isn't reverted on Save; "Select all" selects only the visible/filtered shelf; the delete-collection confirm no longer renders under the Manage dialog; a stale reconcile run can no longer demote a freshly-seeded book to content-absent (cancellation guard + transactional insert); the import path stamps `coverVersion` so a freshly imported cover isn't dropped and re-downloaded; and Escape no longer both closes a dialog and drops the multi-selection. Low-severity sync edge cases (LWW loser convergence, offline account-switch) and UX papercuts (paste-to-parse, search using live form terms) also fixed. A dedicated data-integrity pass over the reconcile/seed/cover machinery found no persistent-corruption defect. Two known-lows are deferred: marking a status bumps the "Recent" sort (needs a separate recency field), and cover *replacement* isn't atomic across the R2 upload + `attachFiles` stamp (an interrupted replace can show a stale cover on other devices until a retry — inherent to the fixed cover key).
